DIY Storage: Fundamentals

Serge, 
November 20th, 2019 · 9 min read

The biggest dichotomy in software is arguably code vs data. It all goes back to von Neumann vs Harvard architecture, where the major debate was about whether we keep code and data separately (more explicit, easier to understand and reason about) or in shared memory (cheaper). Cheaper option won as it always does and so our industry spent more than half a century applying band aids to it. Right now it’s mostly stabilized around logical separation since physical is unattainable. This distinction pops up all over the field all the time. Here are some examples:

  • First two services that Amazon released in AWS were S3 (storage aka data) and EC2 (compute aka code). Similar pattern occurred in Azure and GCP rollouts.
  • To this day all modern computers have two major compute components (CPU and GPU) and two “storage” components (Random Access Memory and Mass Storage — hdd/ssd). RAM is often lumped with “compute”, but not because it does any computations on its own (it most definitely is storing data), but because CPU is absolutely useless without it. All other peripherals are mostly irrelevant commodity.
  • There is a special term for programming languages that don’t really care about this distinction.
  • Remote code execution is data (arrived on the machine over network) ending up treated as code (executed), SQL injection is another class of attacks that are based on data being confused for code.
  • One of The Great Debates of modern times (along with “tabs vs spaces” and “emacs vs vim”) is embedding code in data or data in code.

Now, why did I tell all of that? Mostly to show off, but also to put another concept into a spotlight: mutability. In all seriousness, though, reasoning about code is hard enough without it morphing and shapeshifting while we are trying to do it, so these days our industry mostly abandoned the idea of messing with code while it’s running (except for some very specific cases). This should not be confused with code being changed over time, which is also difficult but we are kinda stuck with having to change things over time, so “deal with it”. On the other hand, something needs to be changed during runtime in order for program to be useful. Anything that a static code could produce by operating on a predefined static data could be calculated during compilation and we wouldn’t need a program anymore - just deliver precomputed result to the user and be done with it. Problem is, those pesky users have their own ideas about what should be calculated over what inputs so we have to figure out how to deal with “variable” data or as they call it “mutable state”. In essence, “storage” is all about saving data (both inserting new pieces of information and changing existing ones), querying it back and deleting what no longer needed. These basic operations are often referred to as CRUD. Despite seeming simplicity they describe a whole world of different requirements. “Read” may mean retrieve information sequentially on a single machine (e.g. restoring backup), or it could mean running a complex query over petabytes of data, or run thousands such queries per second with strict requirement on latency. “Create” may also be as simple as just appending some “bag-of-bytes” at the end of some “stream” or it may mean making decisions on how many copies, distributed around the world do we want to be confirmed before we actually consider “create” operation complete, what happens if client crashes and/or cancels while operation is in progress, at what point this new data gets available for reading and to whom, what happens if multiple clients try to update same piece of data simultaneously (by some definition of simultaneously which is not as obvious as it may seem). Update/delete may be outright banned (it’s more common than one might think) or we may need to store all the different versions of information plus all the goodness of distributing these versions across multiple machines.

In the coming months I’ll explore “mass storage” technologies from filesystems (ntfs, zfs) to “tiled” file storage (hdf5, parquet) to embedded DBs (rocksdb, LiteDB) to standalone DBs (generic SQL DB with data stored in pages, transaction log, replication, secondary indices and so on, I’m not going to cover parsing of actual SQL — that’s big enough can of worms for entire separate series, neither will I try to cover entire SQL standard — not 2016 not any of the earlier ones), to full text search (lucene), to distributed systems (consensus, distributed transactions, consistency models, load balancing, high availability and failover, as well as specifics of some of the popular systems like redis, hdfs, mongodb, kafka and cassandra). I’m fairly well versed in most of these technologies, but not all of them so I’ll need to do some figuring out along the way. That’s in addition to figuring out how to make reimplementation simple while still keeping the spirit/design of the original.

Before I start let me quickly run through some of the basics.

Data by access patterns

  • Readable stream: you can read your data one-by-one, but you can never go back and reread anything plus you can’t write (something else wrote and generated this data: think receiving broadcast TV or looking at stars through a telescope — you can’t exactly go and tell supernova to explode again). At some point stream becomes so big, that we start calling it a “data river”.
  • Writable stream: you can send your data somewhere, and the fact of sending gets acknowledged, but you don’t know what happens to that data and just as with readable streams, you can never go back and redo your mistakes.
  • Read/Write streams: there is usually no such thing as read/write streams. Even when you are able to both read and write (think appending data to the end of file while someone else reads all those appends), it’s not useful to “either read or write” at a current position and after either of operations is completed — advance that current position, so even when we do have an ability to both read and write — these are usually two separate streams with their own independent positions.
  • Rewindable stream: your regular stream, but you can start-over and start consuming the stream from the very beginning.
  • Bidirectional “stream”: not a stream anymore since you can read or write it in both directions, so it is commonly referred to as a “tape” (early computing had a lot of actual physical tapes “flying” around and even to this day magnetic tapes remain the most cost-effective way of keeping backups).
  • Random access: as the name suggests, you can access your data at any position any time you want. Similar to the concepts above, ability to read and/or write is independent of where we can access our data.

Data by internal structure

Unstructured data

Simple pile (or bag) of bytes without any apparent structure. Usually there is some underlying structure, but it’s not obvious. It could be anything from first bajillion digits of pi to raw recording of some natural phenomena (including but not limited to white noise) to any data encrypted with a stream cipher. This type of storage has pretty limited usefulness. Also in this category data that may have structure, but there is absolutely no guarantee that it does: files on the filesystem, blobs in the DB, pipes between processes.

Basic structure

Simplest form of structured storage is a stream of primitive data. It doesn’t need to be delimited since every entry has a fixed size. Block ciphers come to mind — you wouldn’t be able to discover the structure from observing data alone, but it is there and it becomes apparent from analyzing algorithm that produces this data. I cannot really think of anything else that would fit here.

Record stream

From a stream of primitives let’s move on to the stream of some complex entries called records. This is a two-tier (network protocol stacks have way more than two tiers but idea is similar) structure: first we need to separate records from each other and then, we can interpret data within each record. This gives us a “log” — stream of records where only two operation are available: read stream from the start one record at a time and append one record to the end of the stream (plus sometimes third operation is provided: truncating start of the log to a specific position).

There are two ways to delimit entries: in-band and out-of-band. In-band is simple: think comma-separated values. Both records and subfields within records are separated using control characters (newlines and commas respectively). At any point if we observe a “newline” control character, we end current record and open a new one. But what if we have newlines in our data? OK, let’s introduce special quotation syntax. But what if we have quotes in our data? Here’s a special syntax to use quotes to “escape” other quotes. It gets really messy really fast. In case of CSV it becomes even messier because there is no agreement or standard on how these special cases must be treated so we have a dozen or so slightly different (but enough to be incompatible) versions of it. Out-of-band is more robust — it guarantees by construction that there is no confusion between data and control information. Notice that even if you use separate TCP stream to send data — it’s still same packet stream on network (IP) layer and below, if you store control data in a separate file on a filesystem — it is still stored on the same physical disk. Separation doesn’t have to by physical (and what is physical separation anyway?), your format must guarantee that at any point of time you expect either metadata or data. For example, if we prepend each record with its length in bytes (which itself is fixed length), after we read this length, we switch into a “data” mode and for the next “length” bytes we don’t put any additional interpretation on what we read, then after “length” bytes of uninterpreted data are received, we switch back to expecting length and so forth.

For all its simplicity this concept is widely used all over the place: from performance monitoring, to error logging, to redo/transaction logs, to file archives, to almighty MapReduce (yes, yes it’s not the most hyped tech anymore) which doesn’t need anything but stream of records and, in fact, often used over “clickstreams” — unstructured records of user activity.

Random access bag of records

Sometimes reading all records one by one is not desirable and we need to be able to access any record as efficiently as possible. Ever noticed that in some compressed files you can quickly unpack any file inside it no matter how big the entire archive is and in some — you need to wait for minutes in order to unpack tiny 100-bytes text file? Same principle applies to other data as well — in some situations you don’t need all your information at once and want to pick and chose instead. This is achieved using lookup tables (or indices — similar but separate concept from database secondary indices): remember those record lengths from previous chapter? They all have same length, why don’t we extract them all and store them separately with small modification: instead of length — let’s store start offset into the raw data (stored separately). Length is trivially calculated as a difference between two adjacent offsets. This way we can immediately jump to any record by first finding its offset in the LUT (simple arithmetic: LUT consists of entries of size S, so offset to zero-based entry N in LUT is N * S) reading LUT at this offset gives us position of our record. Updating any of the records would require overwriting all the data that follows that record as well as recalculating all offsets of the affected records, but we can easily append records. Depending on our needs we can even avoid having to have two files — one with LUT and one with raw data: just store data, then LUT, then a single number that tells offset of the LUT. This way we can trivially find both data and LUT as well as add records without overwriting entire file. In fact some formats (e.g. pdf) are using storage scheme similar to this — they store some of the metadata in footer instead of header and it allows them to progressively “stream” pdf files to users as well as allow for incremental updates without overwriting everything. This is similar to book’s table of contents, but placed at the end rather than the beginning of the book.

Dictionaries

Accessing data using some numeric index is fun, but not always useful. More often we want to be able to find data using some label or “key”. Think about phone book… scratch that — no one uses those anymore. OK, dictionary it is then. Physical paper dictionary has all terms in it sorted alphabetically to allow for efficient search of any term, once the term is found definition of that term can be anything - arbitrary text, pictures or even references to other definitions and/or external sources.

Let’s ignore meta blocks for a moment and what we see is data blocks are followed by variable size index block (with one key to offset/size mapping for every data block) and fixed size footer that has offset to index. Looks almost primitive, though if we look where it is used we might get surprised. Sstables (sorted string tables — files that are described above) originated from Google’s BigTable paper and leveldb although the idea of immutable sorted files that are sometimes merged together into a new set of immutable files (old files are deleted right after) instead of in-place updates comes from LSMT paper. Sstables are now used in data-heavy applications in Google, Facebook (they have tendency to put rocksdb into everything — MyRocks, Rocksandra, LogDevice, etc and for a good reason: rocksdb is awesome), and basically everywhere since sstables are used in Kafka, Cassandra, Flink, Ceph, etc..

Guess what dictionary data structure is used even more than sstable? A tree. In particular in case of external storage, B-trees are basically everywhere - and that’s what we’ll look into in the next post. This time with actual code, I promise.
